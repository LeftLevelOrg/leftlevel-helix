# Product Differentiation

This document explains why LeftLevel can be useful and recognizable even with an open-source core.

## Product focus

LeftLevel should win by being trustworthy, simple, transparent, and pleasant to use.

The goal is not to hide the protocol. The goal is to build the clearest and most usable product around it.

## Differentiators

| Differentiator | Why it matters |
|---|---|
| No phone number or email identity required | Users can create protected connections without joining a central account directory. |
| Server-blind relay model | The relay does not need plaintext, contact names, or permanent conversation IDs. |
| Pairwise identity | A contact relationship can exist without a global public username. |
| Clear trust labels | NEW, OK, and CHANGED states make verification understandable. |
| Encrypted local app store | Contacts, session state, and history stay encrypted locally. |
| Post-quantum-ready direction | The project starts with hybrid key agreement as a first-class design goal. |
| Honest protection matrix | Users can see what is protected, reduced, and out of scope. |
| Official app plus open core | Trust comes from transparency and execution. |

## Why users may care

Users may choose LeftLevel because they want:

- protected contact setup without phone numbers;
- a communication tool that does not center on a global account;
- a small app focused on trusted communications;
- clear verification status;
- official relays or self-hosted relays;
- transparent security documentation.

## Sustainable project advantages

Open-source products can still build durable advantages through:

1. strong brand trust;
2. official builds;
3. hosted relay reliability;
4. community momentum;
5. documentation quality;
6. review history;
7. app-store presence;
8. support and managed deployments;
9. strong user experience;
10. clear governance.

## Brand and official releases

The code license and the LeftLevel brand should be handled separately.

The open-source license covers code use. The LeftLevel name, logo, official releases, domains, and hosted services should be protected as project assets.

## Commercial path

A healthy open-source commercial path can include:

- hosted relay service;
- paid official builds and support;
- managed protected deployments;
- enterprise setup help;
- training and documentation packages;
- future integrations after the standalone app is stable.

## Recommendation

Build the best official experience first. Let the open-source core create trust, and let the product, brand, support, hosting, and documentation create long-term value.
