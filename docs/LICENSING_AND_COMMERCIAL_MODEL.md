# Licensing and Commercial Model

This document explains the current license status and practical commercial options for LeftLevel Helix.

This is not legal advice. Before accepting substantial outside contributions or launching a commercial product, consult an attorney familiar with open-source licensing.

## Current license

The repository currently uses the MIT License.

MIT is a permissive open-source license. It generally allows people to use, copy, modify, distribute, sublicense, and sell copies of the software as long as the copyright and license notice are preserved.

## What MIT gives the project

- Low friction for contributors and adopters.
- Easy use by companies and developers.
- Attribution through the copyright notice.
- Permission to sell software and services built with the code.
- Broad compatibility with other projects.

## What MIT does not guarantee

- It does not force users to publish their modifications.
- It does not prevent companies from using the code in proprietary products.
- It does not create a built-in revenue stream.
- It does not provide the same explicit contributor patent grant as Apache-2.0.
- It does not protect the LeftLevel brand name or logo by itself.

## Monetization options that work with open source

LeftLevel can remain open source and still make money through:

1. Hosted relay services.
2. Paid desktop/mobile convenience builds.
3. Enterprise support.
4. Security review and deployment consulting.
5. Managed private relay deployments.
6. Training and documentation packages.
7. Commercial integrations.
8. Sponsorships and donations.
9. Paid verification, compliance, or admin features.
10. Open-core features, if the project chooses that route later.

## Credit and attribution

The MIT License preserves copyright notices, but it does not guarantee prominent marketing credit in every product built from the code.

For stronger credit expectations, the project can also maintain:

- a `NOTICE` file;
- brand guidelines;
- contributor recognition in README/docs;
- release notes crediting contributors;
- a website contributors page.

## Brand and trademark

Open-source licensing covers code copyright. It does not automatically grant permission to use the LeftLevel name, logo, domain, or branding.

The project should keep brand usage separate from code licensing. Suggested policy:

- forks may use the code under the license;
- forks may not imply they are official LeftLevel releases;
- the LeftLevel name and logo require permission for commercial use or official endorsement claims.

## Alternative license options

### Keep MIT

Best for adoption and simplicity. Good if monetization comes from hosted services, support, commercial builds, and brand trust.

### Switch to Apache-2.0 before major outside contributions

Apache-2.0 is also permissive but includes a more explicit patent grant and NOTICE handling. This may be preferable for a security protocol project.

### Dual license later

The project may keep a community edition open source while offering commercial licenses or proprietary enterprise features. This requires careful contribution agreements and legal planning.

### Contributor License Agreement or Developer Certificate of Origin

Before significant outside contributions, the project should choose whether to use:

- a Developer Certificate of Origin sign-off;
- a Contributor License Agreement;
- or simple inbound-equals-outbound contribution terms.

## Recommended near-term approach

For now:

1. Keep MIT temporarily to avoid churn.
2. Add clear contribution terms.
3. Add brand and commercial-model docs.
4. Add a `NOTICE` file crediting LeftLevel and contributors.
5. Decide whether to switch to Apache-2.0 before accepting major external contributions.
6. Do not accept large outside code contributions until the license decision is finalized.

## Recommended commercial path

The cleanest path is:

- open-source core protocol and app;
- paid hosted relay service;
- paid support and deployment help;
- paid enterprise/admin features later;
- clear public security posture;
- brand-protected official builds.

This lets the community trust the code while still giving LeftLevel a business path.
