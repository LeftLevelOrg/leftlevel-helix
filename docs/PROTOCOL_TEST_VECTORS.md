# Protocol Test Vectors

This document defines how LeftLevel Helix should create and use protocol test vectors.

## Purpose

Test vectors help future implementations prove they match the reference implementation.

They are especially important before:

- porting the core to Rust;
- building the Tauri app shell;
- accepting external implementations;
- requesting expert review.

## What a vector should prove

A useful vector should prove:

1. Invite shape is stable.
2. Response shape is stable.
3. Transcript hash is stable.
4. Safety number is stable.
5. Session export/import shape is stable.
6. Envelope shape is stable.
7. Visible relay header stays minimized.
8. Decryption produces the expected plaintext.
9. Replay protection behavior is stable.
10. Out-of-order receive behavior is stable.

## Current challenge

The current prototype uses fresh randomness for identities, handshakes, message padding, and nonces. That is correct for real use, but it makes fixed test vectors harder.

The project should not weaken runtime randomness just to make examples easier.

## Recommended approach

Use two classes of vectors:

### 1. Runtime invariants

These are tests that generate fresh data and assert important properties.

Examples:

- ciphertext does not contain plaintext;
- visible envelope header does not contain direction;
- visible envelope header does not contain padding settings;
- replay is rejected;
- out-of-order delivery works within the receive window.

### 2. Deterministic review fixtures

These are created with a test-only deterministic fixture mode. They should live under `tests/fixtures/` and must be clearly marked as not for production.

A fixture can use test-only fixed identities and test-only key material so reviewers can reproduce exact outputs.

## Fixture file layout

Recommended future layout:

```text
tests/fixtures/v0_6/
  alice_identity.json
  bob_identity.json
  invite.json
  response.json
  alice_session_start.json
  bob_session_start.json
  envelope_001.json
  alice_session_after_send.json
  bob_session_after_receive.json
  expected.json
```

## Expected manifest

`expected.json` should include:

```json
{
  "protocol": "LLH-HELIX-v0.2",
  "package_version": "0.6.0",
  "visible_header_keys": ["mailbox_id", "mode", "v"],
  "safety_number": "example-only",
  "plaintext_001": "example-only",
  "notes": "Fixture values must be generated by test-only deterministic helpers, not runtime production randomness."
}
```

## Rules

- Never use deterministic fixtures in production code paths.
- Never publish real user vaults, identities, invite files, responses, session files, or message bodies.
- Clearly mark fixture identities as test-only.
- Keep vectors versioned by protocol version.
- Update vectors only when the wire format intentionally changes.

## Next implementation step

Add a deterministic fixture generator under:

```text
scripts/generate_protocol_vectors.py
```

The generator should write review fixtures under:

```text
tests/fixtures/v0_6/
```

CI should then verify that current code can read the fixture and reproduce the expected behavior.
